# Style Anchors Collection - Document Specifications

## Overview
This document defines the specifications for style anchor documents generated by the style-anchors-collection skill in the Sherpy workflow.

## Document Types
1. **index.yaml** - Master index of all style anchors with metadata and cross-references
2. **[anchor-id].md** - Individual markdown files documenting specific code patterns

---

## 1. index.yaml Specification

### Purpose
Provides a structured index of all collected style anchors, categorizes them by type, and creates a usage matrix mapping technical requirements to applicable anchors.

### Location
`{base_directory}/artifacts/style-anchors/index.yaml`

### Dependencies
- **Input:** `technical-requirements.yaml` (determines pattern categories)
- **Output Used By:** `implementation-planner` (references anchors in task instructions)

### Schema Definition

```yaml
version: string                           # Spec version (1.0.0)
project: string                          # Project name
generated: string                        # ISO 8601 timestamp
technical_requirements_ref: string       # Relative path to tech requirements

categories:                              # Logical groupings of anchors
  - id: string                           # Unique category ID (kebab-case)
    name: string                         # Human-readable name
    description: string                  # What patterns this category covers
    anchors: string[]                    # Array of anchor codes in this category (e.g., ["SA-001", "SA-002"])

anchors:                                 # List of all style anchors
  - code: string                         # Unique anchor code (required, format: SA-NNN)
    id: string                           # Anchor ID (kebab-case, matches filename)
    name: string                         # Display name
    file: string                         # Relative path to .md file
    category: string                     # Category ID reference
    source:                              # Where the code example comes from
      repository: string                 # "current-project" or URL
      path: string                       # File path in repo
      lines: string                      # Line range (e.g., "10-50")
    applies_to:                          # When to use this anchor
      - milestone_types: string[]        # Milestone categories
      - file_patterns: string[]          # Glob patterns
    tags: string[]                       # Search/filter tags

usage_matrix:                            # Maps tech decisions to anchors
  architecture_patterns:                 # From technical requirements
    - pattern: string                    # Architecture pattern name
      anchors: string[]                  # Relevant anchor codes (e.g., ["SA-001", "SA-003"])

  task_types:                            # By task type
    - type: string                       # code|test|docs|config
      recommended_anchors: string[]      # Anchor codes (e.g., ["SA-001", "SA-004"])

meta:                                    # Collection metadata
  total_anchors: number                  # Count of anchors
  collection_date: string                # When collection completed
  last_updated: string                   # Last modification timestamp
  collector: string                      # "interactive" or tool name
```

### Validation Rules

#### Required Fields
- `version`, `project`, `generated`, `technical_requirements_ref`
- `categories` must contain at least 1 category
- `anchors` must contain at least 1 anchor
- Each anchor must have: `id`, `name`, `file`, `category`, `source`, `applies_to`, `tags`

#### Field Constraints
- `version`: Must be valid semver (e.g., "1.0.0")
- `generated`: Must be ISO 8601 format
- Category `id`: Must be unique, kebab-case, alphanumeric+hyphens only
- Anchor `code`: Must be unique, format SA-\d{3} (e.g., SA-001, SA-042)
- Anchor `id`: Must be unique, kebab-case, alphanumeric+hyphens only
- Anchor `code` and `id` must match filename (SA-{code}-{id}.md)
- Anchor `category`: Must reference existing category ID
- Source `lines`: Must be format "N" or "N-M" where N < M
- `file_patterns`: Must be valid glob patterns
- `tags`: All lowercase, kebab-case

#### Cross-Field Validation
- All anchor codes in `categories[].anchors` must exist in `anchors[]`
- All `anchors[].category` must reference valid `categories[].id`
- All `anchors[].file` must reference existing .md files matching SA-{code}-{id}.md pattern
- All anchor codes in `usage_matrix` must exist in `anchors[]`
- Anchor codes are used for references (SA-001), IDs are for filenames

#### Common Errors to Prevent
- Duplicate anchor codes (SA-001 used twice)
- Duplicate anchor IDs
- Code/ID mismatch with filename
- Orphaned anchors (not in any category)
- Missing .md files referenced in index
- Invalid line range format
- Category references that don't exist
- Empty categories or usage_matrix

### Generation Guidelines

1. **Category Identification**
   - Parse technical-requirements.yaml for key patterns
   - Create categories based on architecture sections
   - Typical categories: services, testing, validation, api, data-access, error-handling

2. **Interactive Collection**
   - For each category, ask user to identify 1-3 exemplar files
   - Prompt: "For [category], what's your best example of [pattern]?"
   - Collect: repository, file path, line range, description

3. **Usage Matrix Generation**
   - Map architecture patterns from tech requirements to anchors
   - Associate task types with recommended anchors
   - Ensure every anchor is referenced in usage matrix

4. **Metadata Population**
   - Count total anchors automatically
   - Set collection_date to current timestamp
   - Mark collector as "interactive" or "automated"

### Usage Example

When implementation-planner generates tasks, it:
1. Loads index.yaml
2. Filters anchors by milestone type and file patterns
3. Includes relevant anchor codes (SA-001, SA-003) in task instructions
4. Developers look up SA-*.md files for detailed patterns
5. Milestones.yaml can reference global anchors by code: `style_anchors: [SA-001, SA-003]`

## 2. [anchor-id].md Specification

### Purpose
Documents a specific code pattern with example code, usage guidelines, common mistakes, and references to related anchors.

### Location
`{base_directory}/artifacts/style-anchors/[anchor-id].md`

### Naming Convention
- Filename matches anchor `id` from index.yaml
- Kebab-case (e.g., `service-pattern.md`, `test-effect-services.md`)
- Extension must be `.md`

### Schema Definition

```markdown
---
id: string                     # Must match filename and index.yaml entry
name: string                   # Display name
category: string               # Category ID from index.yaml
tags: string[]                 # Same as in index.yaml
created: string                # YYYY-MM-DD format
updated: string?               # YYYY-MM-DD format (optional)
---

# [Name]

## Overview
[2-3 sentence description of what this pattern demonstrates]

## Source Reference
**Repository:** [current-project or URL]
**File:** `[path/to/file.ext]`
**Lines:** [N-M]

[Optional: Link to source if available]

## Code Example
```[language]
// [file:lines] or [description]
[Actual code snippet - 20-50 lines typical]
```

[Optional: Additional code examples for variations]

## What This Demonstrates
[Bulleted list of key concepts shown in the code]
- [Point 1]
- [Point 2]
- [Point 3]

## When to Use
[Clear guidance on when this pattern applies]
- [Scenario 1]
- [Scenario 2]
- [Scenario 3]

## Pattern Requirements
[Mandatory rules when using this pattern]
✓ [Requirement 1 - use checkmark for must-do]
✓ [Requirement 2]
✓ [Requirement 3]

## Common Mistakes to Avoid
[Anti-patterns and pitfalls]
❌ [Mistake 1 - use X for don't-do]
❌ [Mistake 2]
❌ [Mistake 3]

## Related Anchors
[Cross-references to complementary patterns]
- `[anchor-id]` - [Brief description of relationship]
- `[anchor-id]` - [Brief description of relationship]

## Test Coverage
[Optional: Reference to corresponding test pattern]
See `[test-anchor-id].md` for testing this pattern.

## Additional Notes
[Optional: Edge cases, performance notes, version-specific notes]
```

### Validation Rules

#### Required Sections
- YAML frontmatter with `id`, `name`, `category`, `tags`, `created`
- `# [Name]` heading matching frontmatter
- `## Overview`
- `## Source Reference`
- `## Code Example`
- `## What This Demonstrates`
- `## When to Use`
- `## Pattern Requirements`
- `## Common Mistakes to Avoid`

#### Optional Sections
- `## Related Anchors`
- `## Test Coverage`
- `## Additional Notes`

#### Content Constraints
- Overview: 2-3 sentences, concise
- Code Example: 20-50 lines (can be longer if justified)
- Code must include language identifier in fence
- Must use ✓ for requirements, ❌ for mistakes
- Pattern Requirements: 3-7 checkmarked items
- Common Mistakes: 3-7 cross-marked items

#### Frontmatter Validation
- `code` must match pattern SA-\d{3} (e.g., SA-001)
- `code` must match filename prefix
- `code` must exist in index.yaml
- `code` must be unique across all anchors
- `category` must match category in index.yaml
- `tags` must match tags in index.yaml
- `created` must be YYYY-MM-DD format
- `updated` if present, must be >= `created`

### Generation Guidelines

1. **Interactive Data Collection**
   - Ask user: "What file demonstrates [pattern]?"
   - Ask user: "What line range? (format: 10-50)"
   - Ask user: "What does this code demonstrate?"
   - Ask user: "When should developers use this pattern?"

2. **Frontmatter Generation**
   - Assign next sequential `code` (SA-001, SA-002, etc.)
   - Generate `id` from pattern name (kebab-case)
   - Set filename as SA-{code}-{id}.md
   - Set `created` to current date
   - Set `category` from parent category selection
   - Set `tags` based on pattern characteristics

3. **Code Example Extraction**
   - If source is accessible, read specified lines
   - Include file:line reference as comment
   - Preserve indentation and formatting
   - Add syntax highlighting language identifier

4. **Requirements and Mistakes**
   - Extract from best-practices knowledge
   - Align with technical-requirements constraints
   - Use consistent formatting (✓ and ❌)

5. **Cross-References**
   - Link to related patterns (e.g., service → test pattern)
   - Reference complementary anchors
   - Create bidirectional relationships where applicable

### Usage Example

When a developer works on a task:
1. Task instructions reference anchor code (e.g., "Follow **SA-001**")
2. Developer opens SA-001-service-pattern.md file
3. Reviews code example and requirements
4. Follows pattern when implementing
5. Checks "Common Mistakes" before committing
6. Cross-references related anchors by code (**SA-004** for testing)

## Integration with Workflow

### Workflow Position
**Step 4: Style Anchors Collection** (between Technical Requirements and Implementation Planning)

```
Step 3: Technical Requirements → technical-requirements.yaml
         ↓
Step 4: Style Anchors Collection → style-anchors/index.yaml + *.md files
         ↓
Step 5: Implementation Planner → milestones.yaml + tasks (references anchors)
```

### Data Flow

**Inputs:**
- `technical-requirements.yaml` (determines what patterns to collect)
- User-provided file paths and line ranges
- Existing codebase or reference repositories

**Outputs:**
- `style-anchors/index.yaml` (master index)
- `style-anchors/*.md` (individual anchor documents)

**Consumers:**
- Implementation Planner (embeds anchors in task instructions)
- Implementation Plan Review (validates anchor usage)
- Developers (reference during implementation)

## Example Scenarios

### Scenario 1: New Greenfield Project
- Few or no style anchors to collect
- May reference external repositories (Effect examples, framework patterns)
- Focus on establishing conventions early

### Scenario 2: Existing Codebase
- Identify best existing files as anchors
- Document current patterns before extending
- Use for consistency in new features

### Scenario 3: Multi-Pattern Project
- Multiple architecture layers (API, services, data)
- Collect 1-2 anchors per layer
- Create comprehensive usage matrix

## Quality Checklist

Use this checklist when generating style anchor documents:

### index.yaml
- [ ] All required fields present
- [ ] Valid semver version number
- [ ] All anchor IDs are unique
- [ ] All category references resolve
- [ ] Usage matrix covers all anchors
- [ ] File references match actual .md files
- [ ] Line ranges are valid format
- [ ] Tags are consistent and useful

### Individual .md Files
- [ ] Frontmatter matches index.yaml entry
- [ ] All required sections present
- [ ] Code example is clear and complete
- [ ] Pattern requirements are actionable
- [ ] Common mistakes are specific
- [ ] Related anchors create useful connections
- [ ] Formatting is consistent (✓ and ❌)
- [ ] Language identifier on code fence

### Overall Collection
- [ ] Covers key patterns from technical requirements
- [ ] Balanced across categories (not all in one)
- [ ] Examples are exemplary, not just "any code"
- [ ] Cross-references form useful network
- [ ] No orphaned anchors (not in any category)
- [ ] File naming is consistent (kebab-case)
